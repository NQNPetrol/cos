## Descripción

Dos problemas en la sección de rodados de administración:

1. **Error 413 / 500 al adjuntar factura o comprobante** en `/rodados/pagos/{id}/adjuntar-factura`. Nginx devuelve `413 Request Entity Too Large` porque `client_max_body_size` no está configurado (default 1 MB). Después aparece un 500 genérico. Laravel valida hasta 10 MB pero nginx corta antes. Se necesita: (a) instrucciones para configurar nginx, y (b) manejo amigable de `PostTooLargeException` para que no aparezca pantalla de error genérica.

2. **Columna Estado en la tab "Servicios"** de `/rodados` siempre muestra "Pendiente". La migración define `estado` como `enum('pendiente', 'completado')` con default `pendiente`, pero nada cambia el estado automáticamente. Se necesita rediseñar la lógica de estados para que sean dinámicos según la fecha del turno y la confirmación del usuario.

## PASO 3 — Migración: rediseñar estados de turnos

Crear una nueva migración que modifique la columna `estado` de `turnos_rodados`:

1. Cambiar de `enum('pendiente', 'completado')` a `string` (más flexible para los nuevos estados) o a un `enum` con los valores correctos.
2. Los estados válidos son:
   - `programado` — turno creado, fecha futura (>7 días)
   - `proximo` — faltan 7 días o menos (este estado es **calculado**, no se guarda en DB)
   - `asistido_a_confirmar` — fecha del turno ya pasó, no se confirmó (este estado es **calculado**, no se guarda en DB)
   - `asistido` — usuario confirmó asistencia
   - `cancelado` — usuario canceló
   - `perdido` — usuario confirmó que no asistió
3. Migrar datos existentes: todos los `pendiente` → `programado`. Los `completado` → `asistido`.
4. Actualizar constantes en `TurnoRodado.php`:

```php
const ESTADO_PROGRAMADO = 'programado';
const ESTADO_ASISTIDO = 'asistido';
const ESTADO_CANCELADO = 'cancelado';
const ESTADO_PERDIDO = 'perdido';
```

5. Agregar un accessor o método en el modelo para calcular el estado visual:

```php
public function getEstadoVisualAttribute(): string
{
    if (in_array($this->estado, ['asistido', 'cancelado', 'perdido'])) {
        return $this->estado;
    }
    // Estado es 'programado' — calcular según fecha
    if ($this->fecha_hora && $this->fecha_hora->isPast()) {
        return 'asistido_a_confirmar';
    }
    if ($this->fecha_hora && $this->fecha_hora->diffInDays(now()) <= 7) {
        return 'proximo';
    }
    return 'programado';
}
```

**Verificación:** `php artisan migrate` pasa. Datos existentes migrados correctamente. El accessor devuelve el estado visual correcto según la fecha.

## Notas

- La configuración de nginx (`client_max_body_size`) debe aplicarse manualmente en el servidor de producción. El prompt genera la documentación; el deploy lo hace el usuario.
- Los estados "próximo" y "asistido_a_confirmar" son **calculados** (accessor), no se guardan en DB. Esto evita necesitar un cron/scheduler para actualizar estados.
- La confirmación de asistencia no es obligatoria — si el usuario no confirma, queda como "asistido (a confirmar)" indefinidamente.
