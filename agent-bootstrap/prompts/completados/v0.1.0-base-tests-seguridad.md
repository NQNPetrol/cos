# v0.1.0 — Base de tests + seguridad

**VERSIÓN:** v0.1.0  
**SLUG:** base-tests-seguridad  
**DEPENDENCIAS:** ninguna  
**ESTIMACIÓN:** 1–2 días  
**PRIORIDAD:** alta

---

## Descripción

Dejar lista la **base de testing** del COS y agregar tests de **seguridad y autenticación**: login, logout, rutas protegidas, y permisos por rol (admin vs cliente) con Spatie. El proyecto ya tiene Pest, PHPUnit, SQLite en memoria, `RefreshDatabase` en Feature, y un workflow de CI que ejecuta `./vendor/bin/pest`. Esta versión consolida esa base y añade tests de auth y autorización para que cualquier cambio en permisos o rutas quede cubierto.

Sin esta base, los tests de módulos (v0.2.0) no tienen piso consistente.

## PASO 2 — Tests de autenticación

En `tests/Feature/Auth/` (o equivalente):

- Test: pantalla de login se renderiza (GET `/login` → 200).
- Test: usuario puede autenticarse con email y contraseña correctos; redirección según rol (dashboard admin o client según corresponda).
- Test: usuario con credenciales incorrectas no se autentica (mensaje de error o 422, sin sesión).
- Test: usuario autenticado puede cerrar sesión (POST `/logout` o la ruta que use la app); después queda como guest.

Usar `User::factory()` para crear usuarios; contraseña la que use la factory (ej. `password`). Si la app usa Livewire para login (como en `AuthenticationTest.php` existente), mantener ese estilo.

**Verificación:** `./vendor/bin/pest tests/Feature/Auth` pasa.

## PASO 3 — Tests de autorización (permisos)

Crear tests que verifiquen que las rutas respetan permisos Spatie:

- Test: usuario **sin** permiso no accede a una ruta protegida. Ejemplo: usuario con rol que no tiene `ver.eventos` (o `ver.eventos-cliente`) al hacer GET a la ruta de listado de eventos recibe 403 o redirección a login / “no access”, no 200.
- Test: usuario **con** permiso accede. Ejemplo: usuario con rol admin (o un rol con `ver.eventos`) hace GET al listado de eventos admin y recibe 200.
- Test: al menos un flujo **cliente**: usuario con rol cliente (o clientadmin) con permiso `ver.eventos-cliente` accede a la ruta cliente de eventos (bajo prefijo `client/`) y recibe 200; usuario sin ese permiso no accede.

Para asignar roles en tests: usar `RolesSeeder` / `PermissionsSeeder` en setUp o en un trait, o crear el rol/permiso en el test con `Role::firstOrCreate`, `Permission::firstOrCreate`, y `$user->assignRole('admin')` (o el rol que corresponda). Usar `RefreshDatabase` para no contaminar datos.

**Verificación:** tests de autorización pasan; al menos un caso admin y un caso cliente cubiertos.
