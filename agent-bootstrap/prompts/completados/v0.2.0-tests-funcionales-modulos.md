# v0.2.0 — Tests funcionales de módulos críticos

**VERSIÓN:** v0.2.0  
**SLUG:** tests-funcionales-modulos  
**DEPENDENCIAS:** v0.1.0 (base de tests + seguridad)  
**ESTIMACIÓN:** 2–3 días  
**PRIORIDAD:** alta

---

## Descripción

Agregar tests **unitarios** y **feature (HTTP)** para flujos críticos del COS en tres módulos: **Eventos**, **Rodados** y **Tickets**. El proyecto ya usa Pest, PHPUnit, SQLite en memoria y `RefreshDatabase`; esta tarea amplía la cobertura a lógica de negocio y rutas de estos módulos, sin tocar integraciones reales con HikCentral/Flytbase/S3 (usar mocks si aparecen en el flujo).

Objetivo: que cambios en esos módulos tengan red de seguridad y que la CI pueda ejecutar la suite completa.

## PASO 1 — Factories y datos de prueba

Verificar o crear factories necesarias para Evento, Rodado y Ticket (y modelos relacionados que usen: Cliente, User, etc.).

- Si existe `database/factories/EventoFactory.php` (o equivalente), revisar que permita crear eventos con estado válido y relaciones mínimas (cliente, categoría si aplica).
- Si no existe, crear factory para `Evento` con atributos requeridos por el modelo y las validaciones del controller.
- Igual para `Rodado` y `Ticket`: tener al menos una factory por modelo que permita crear registros válidos para tests.

**Verificación:** `php artisan tinker` → `Evento::factory()->create()`, `Rodado::factory()->create()`, `Ticket::factory()->create()` (o el nombre de clase que corresponda) sin errores.

## PASO 2 — Tests unitarios de modelos/servicios

En `tests/Unit/`:

- **Evento:** al menos un test que valide reglas de negocio o validaciones importantes (por ejemplo: fechas, estado, atributos requeridos). Si la lógica está en un Service, testear el Service.
- **Rodado:** al menos un test de validación o regla de negocio relevante (ej. relación con cliente, atributos obligatorios).
- **Ticket:** al menos un test de creación o estado (ej. que un ticket tenga estado inicial correcto).

Mantener tests enfocados y rápidos; usar factories donde haga falta.

**Verificación:** `./vendor/bin/pest tests/Unit` pasa (o `php artisan test --testsuite=Unit`).

## PASO 3 — Tests feature: Eventos (admin)

En `tests/Feature/` (crear subcarpeta si se desea, ej. `Eventos/`):

- Test: usuario con permiso `ver.eventos` accede a listado de eventos (GET `/eventos` o la ruta admin correspondiente) y recibe 200.
- Test: usuario con permiso `crear.eventos` puede acceder al formulario de creación (GET create) y enviar POST con datos válidos; se crea un evento y redirección correcta.
- Test: usuario con permiso `editar.eventos` puede editar un evento existente (GET edit, PUT update con datos válidos).
- Test: usuario sin permiso no accede a rutas protegidas (403 o redirección a login según el diseño actual).

Usar usuario autenticado con roles/permisos (Spatie) asignados en el test; `RefreshDatabase` para no contaminar datos.

**Verificación:** `./vendor/bin/pest tests/Feature/Eventos` (o la ruta que se use) pasa.

## PASO 4 — Tests feature: Rodados

En `tests/Feature/Rodados/` (o equivalente):

- Test: usuario con permiso de ver rodados accede al listado (ruta bajo prefijo admin de rodados) y recibe 200.
- Test: creación de un rodado vía formulario (POST) con datos válidos usando factory de Cliente si hace falta; se persiste y redirección correcta.
- Al menos un test más de flujo crítico (edición o listado filtrado), según rutas y tiempo disponible.

**Verificación:** suite de feature de rodados pasa.

## PASO 5 — Tests feature: Tickets

En `tests/Feature/Tickets/`:

- Test: usuario con permiso de ver tickets accede a la vista correspondiente (admin) y recibe 200.
- Test: creación de ticket con datos válidos (formulario cliente o admin según lo que esté implementado) y comprobar que se guarda correctamente.

**Verificación:** suite de feature de tickets pasa.

## Notas

- v0.2.0 asume que v0.1.0 (base de tests + auth/permisos) ya está hecha: estructura Pest, CI básico, tests de login y permisos. Si no está, ejecutar primero ese prompt.
- Priorizar claridad y mantenibilidad: tests legibles y datos via factories.
- No es obligatorio cubrir el 100 % de rutas; el objetivo es módulos críticos con red de seguridad.
- Integraciones externas (HikCentral, Flytbase, S3) quedan fuera de este scope; en el futuro se pueden agregar tests de integración con mocks o contratos.
